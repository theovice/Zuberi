# Qwen3:14b Thinking‐Mode in Ollama

**Soft-switch tokens:** Qwen3’s “thinking mode” is toggled with the special tokens **`/think`** and **`/no_think`** (added at the end of user or system messages)【6†L280-L284】, or by setting the `think` parameter in the Ollama API call【1†L123-L131】【6†L280-L284】. There is **no “<think>” command token** that the model recognizes; injecting a literal `<think>` at the start of a prompt will **not** trigger reasoning mode. In fact, Ollama’s template logic expects only `/think` or `/no_think` (and in practice many front-ends intercept leading `/` as commands)【6†L280-L284】【3†L173-L181】. Thus, putting `"<think>"` in the user prompt does not turn on extended reasoning – it is either ignored or treated as plain text. In short, **Ollama will not honor `<think>` as a soft switch** (it has no effect beyond polluting the text); you must use `/think` or API flags to control thinking.

**Thinking-output in responses:** When thinking is enabled, Qwen3 models produce a reasoning trace enclosed in `<think>...</think>` tags. However, Ollama’s API **separates this trace into a separate field**. According to Ollama docs, the chat response has two fields: `message.thinking` (the chain-of-thought) and `message.content` (the final answer)【2†L45-L53】. In normal use of the **native `/api/chat`** endpoint with `think:true`, the `<think>...</think>` portion does *not* appear verbatim in `message.content`. Instead, the text inside `<think>` is returned in `message.thinking` (often empty if not requested), and `message.content` contains the final answer alone【2†L45-L53】【1†L123-L131】. If you are seeing `<think>…</think>` blocks in the **content** of the response (for example when using the OpenAI‐compatible `/v1/chat/completions` endpoint or if think-mode is misconfigured), it means the thinking trace wasn’t properly separated out. In properly configured Ollama/API usage, the client should only see the answer; the `<think>` tags and reasoning text are kept hidden or split off. 

**Hiding or stripping `<think>` blocks:** If `<think>` tags **do leak into your output**, you can remove them via the model’s Modelfile template. The Qwen3:14B template (see Ollama library) explicitly emits `<think>{{ .Thinking }}</think>` around the reasoning text【16†L104-L112】. To strip thinking entirely, edit the template to remove those lines. For example, delete or comment out the lines that output `<think>{{ .Thinking }}</think>` and the final blank `<think></think>` tail in the assistant branch【16†L104-L112】【16†L133-L141】. (As an alternative, Ollama’s CLI offers a `--hidethinking` flag to drop the reasoning trace, but in a Modelfile you would manually omit those `<think>` tags.) Another approach is to force no-thinking mode by default: e.g. prepend `"/no_think "` to each user message in the template【16†L104-L112】【30†L205-L214】 so the model always receives a `/no_think` instruction. This was demonstrated by a user who created a “no-thinking” variant by adding `"/no_think "` at the start of every user block in the Modelfile【30†L205-L214】. In any case, **there is no built-in Modelfile switch like “strip_thinking”:** you must either toggle off thinking or remove the template’s `<think>` outputs.

**Per-request thinking control:** The supported, reliable way to control thinking per request is to use the Ollama chat API’s `think` parameter. For example, sending `"think": true` (and `"stream": false`) in a `/api/chat` JSON call will enable the thinking mode for that reply【1†L123-L131】, whereas `"think": false` disables it. This matches Ollama’s documentation (and the Qwen3 blog) on enabling reasoning【1†L123-L131】【6†L280-L284】. Another pattern is to append `/think` or `/no_think` to your prompt content (as shown in Qwen3’s examples)【6†L280-L284】. Note that if you use the Ollama **CLI or interactive mode**, `/think` or `/set think` commands toggle the mode. In short, the **native API allows per-call toggling via `think:true/false`**【1†L123-L131】, and the model itself respects `/think` tokens (if properly included in the prompt)【6†L280-L284】. 

**14B-specific issues:** Community reports indicate some quirks with Qwen3:14B. In certain cases, the 14B model has been observed to output only `<think>...</think>` content without ever issuing a final answer or function call【18†L225-L234】【21†L361-L364】. For example, one user noted that in a function-calling scenario Qwen3:14B sometimes never invoked the function and instead responded solely with a (long) thought trace【18†L225-L234】【21†L361-L364】. This appears to be a modeling behavior (possibly due to the logic in its tuned template) rather than an Ollama suppression. No such bug has been reported for the larger Qwen3 models (it seems specific to some runs of the 14B release). The safe practice is to ensure that thinking is explicitly disabled when not needed. If you must use thinking mode, be aware that **14B may occasionally get “stuck” in reasoning blocks**. Monitoring the `message.thinking` output can help detect this – if it grows but `message.content` is empty or repetitive, you may need to adjust the prompt or use `/no_think`. 

In summary, **Ollama’s Qwen3:14B requires using `/think` (or the `think` parameter) to trigger reasoning**; the `<think>…</think>` tags are part of its output format and normally placed in a separate response field, not shown to the user unless misconfigured. To prevent any `<think>` tags from appearing at all, you must modify the model’s template (or use Ollama’s hide-thinking option)【16†L104-L112】【2†L45-L53】. Using the official `/api/chat` interface with the `think` flag is the recommended way to control thinking on a per-request basis.
