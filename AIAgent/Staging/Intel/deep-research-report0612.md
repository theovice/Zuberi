## Ollama API vs OpenAI-Compatible Mode  
- **Native Ollama API:** Ollama’s native API (e.g. `http://localhost:11434/api/chat`) fully supports streaming, function/tool calling, and multimodal inputs.  For example, Ollama’s docs show using `/api/chat` with a JSON body that includes a `tools` array to invoke functions on models like Qwen3【52†L107-L115】.  This direct mode passes our prompt and tool definitions properly to the model, preserving response structure (including assistant “thoughts” and tool outputs).  
- **OpenAI-Compatible (`/v1`) mode:** In contrast, the `/v1` OpenAI API endpoint in Ollama does not handle multi-step tool interactions reliably.  Users report that in `/v1` mode, models often return raw tool JSON as text or ignore tool calls.  For example, one user found that changing from `api: "openai-completions"` to `api: "openai-chat"` (chat format) was necessary for tools to work【54†L571-L579】.  In practice, **all three models** here work best when accessed via Ollama’s native API.  We should update the config to use `baseUrl: "http://host.docker.internal:11434"` (no `/v1`) and set `api: "ollama"` (or omit it), so that OpenClaw uses Ollama’s proper function-calling endpoint. This matches examples where `"endpoint": "http://localhost:11434"` is used【54†L598-L607】.  

## Model Compatibility and Behavior  
- **Gemma 3 (12B):** An open-source Gemini-derived model, Gemma 3 is inherently **multimodal** (text+image)【39†L98-L103】. To use its vision capabilities, we must supply image data in the OpenClaw configuration (e.g. set `input: ["text","image"]`).  Ollama requires version 0.6+ for Gemma3. A Google Cloud article demonstrates that Gemma3 via Ollama can perform function (tool) calls for tasks like web search【41†L58-L63】.  In other words, Gemma3 *can* call tools through Ollama when properly configured. We should verify in staging that giving Gemma3 a tool schema (as shown in [52]) yields a valid function call response.  
- **GPT-OSS (20B):** Ollama’s GPT-OSS:20B is explicitly built for “agentic capabilities”【48†L177-L184】. The Ollama docs emphasize that gpt-oss models support function calling, web browsing, Python calls, and structured outputs【48†L177-L184】. This means GPT-OSS:20B should handle OpenClaw’s tool interfaces natively. In fact, Ollama’s recommended models for OpenClaw include GPT-OSS:20B【37†L58-L64】. We should ensure GPT-OSS is pulled in Ollama and check it returns structured tool JSON (not raw text) in staging.  
- **Qwen 2.5 Coder (14B):** This is a coding-specialist model. Ollama’s community guidance explicitly lists Qwen2.5-coder as a top choice for tool-using agents【46†L78-L87】. The dev.to guide recommends `qwen2.5-coder:32b` (a larger variant) and also highlights GPT-OSS:20B for agents【46†L78-L87】. A Reddit thread noted that even Qwen2.5-7B struggled, but using 14B or above resolves many issues【54†L571-L579】. We should check that the 14B instruct version (as configured) can parse OpenClaw’s tool-laden system prompts. If not, we may need to disable `reasoning` or TTS as a workaround【54†L571-L579】, or switch to 32B if resources allow.  

## Testing Tool-Calling and Multimodal Inputs  
- **Basic tool call test:** For each model, send a known prompt with a simple tool. For example, use Ollama’s example: `model: "<model>"`, a user message like “What’s the temperature in Paris?”, and a tool definition for `get_temperature`【52†L107-L115】. The model should respond with a tool invocation JSON (as in [52]) rather than plaintext.  We should verify this via `curl` or Python (`ollama.chat`) calls, ensuring we get a `tool_calls` entry and valid output.  
- **Gemma3 vision test:** Provide an image input along with text to Gemma3. For instance, add an image (e.g. a city skyline) and ask for description or weather. Gemma3’s multimodality requires including an image in the request. We should test that OpenClaw (via Ollama) correctly routes image payloads to Gemma3. Ollama’s docs show that Gemma3 is invoked with `messages`: [{"role":"user","content":...}] in its examples【39†L24-L32】, but for images we may need a special format or endpoint. The key is confirming the assistant can accept and reason about images.  
- **Stream & role behavior:** Confirm that streaming works on each model. For GPT-OSS and Qwen, the Ollama docs imply streaming (and chain-of-thought) is possible【48†L177-L184】. We should enable streaming to see partial outputs. Also, ensure OpenClaw’s “thinking” messages don’t get misinterpreted (if using `reasoning: true`, Ollama might treat the prompt differently, as noted by community threads【35†L360-L369】). If needed, we can test toggling `reasoning` on/off per model.  

## Recommended Config Changes and Rollout  
- **Switch to native Ollama mode:** In `openclaw.json`, set the Ollama provider `baseUrl` (or `endpoint`) to `http://host.docker.internal:11434` (no `/v1`) and `api: "ollama"`. For example:  
  ```json
  "ollama": {
    "endpoint": "http://localhost:11434",
    "models": ["ollama/gemma3:12b", "ollama/gpt-oss:20b", "ollama/qwen2.5-coder:14b"]
  }
  ```  
  This matches documented configs【54†L598-L607】. Ensure each model’s name matches Ollama’s library (prefix `ollama/` or just the name if auto-detected).  
- **Adjust model lists:** If using multiple models, list them under `agents.defaults.models`. For example, a primary/fallback setup might use GPT-OSS:20B as primary, with Gemma3 and Qwen as fallbacks. (One config example was to put `llama3.3` then `qwen2.5-coder:32b`【46†L78-L87】.) Ensure OpenClaw’s `models list` command shows all intended models.  
- **Disable unsupported modes for small models:** If any smaller Qwen/Gemma variants are in use, consider setting `"reasoning": false` and disabling TTS as one Reddit user did【54†L571-L579】. This ensures the prompt is sent as “system” role instead of “developer” which Ollama may not accept【35†L360-L369】. But with our larger models (12B, 20B, 14B), we expect full reasoning to work.  
- **Rollout steps:** 1) Update `openclaw.json` as above. 2) Restart OpenClaw gateway and verify with `openclaw models list` and `ollama list` to confirm the models are found. 3) Run smoke tests in staging: e.g. have each model perform a simple tool call (as above) and check the output format. 4) Enable each model in a test agent and have it execute a known workflow (e.g. “find a GitHub repo about X” using a search tool).  
- **Regression tests:** Add automated tests covering the key functions. For example: (a) query with a function/tool call on each model (expect a tool invocation in the response). (b) a short multi-turn conversation (say 3–4 turns) to ensure context is preserved. (c) for Gemma3, pass a sample image and verify an image-related answer. Monitor for any language switch or malformations. Keeping targeted tests (rather than a bloated suite) is crucial【54†L571-L579】【29†L57-L64】.
